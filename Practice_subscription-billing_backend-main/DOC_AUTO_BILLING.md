# Сценарий: Ежемесячное автоматическое списание (Работа планировщика)

## 1. АЛГОРИТМ РАБОТЫ

1. Серверный планировщик (Cron или APScheduler) автоматически запускает скрипт проверки каждые сутки в 00:00.

2. Скрипт делает запрос к БД и выбирает все записи подписок, у которых:
   - `status` = "active" (активна)
   - И `next_billing_date` <= Текущая дата-время сервера.

3. ДЛЯ КАЖДОЙ найденной подписки из списка выполнить:
   - Найти стоимость тарифа `price` через связанную таблицу `tariffs`.
   - Вызвать функцию имитации списания денег со счета пользователя на сумму `price`.

   - Проверить результат операции списания:
     - ЕСЛИ списание успешно, ТО:
       1. Обновить дату следующего списания: `next_billing_date` = `next_billing_date` + `period_months` (период тарифа в месяцах).
       2. Создать запись в таблице `payments` со статусом `"paid"`.
     - ИНАЧЕ (ошибка списания / недостаточно средств), ТО:
       1. Изменить статус подписки на `"paused"` (приостановлена).
       2. Создать запись в таблице `payments` со статусом `"failed"`.

4. Завершить работу планировщика и записать лог выполнения.

## 2. ТИПЫ ДАННЫХ

| № | НАЗВАНИЕ ПОЛЯ | ТИП |
| :--- | :--- | :--- |
| 01 | subscription_id | целое |
| 02 | status | перечисление (active, paused, cancelled, trial) |
| 03 | price | дробное |
| 04 | next_billing_date | дата-время |
| 05 | payment_status | перечисление (paid, failed, pending) |
| 06 | period_months | целое |

## 3. ПРЕОБРАЗОВАНИЯ

| ИСХОДНЫЕ ДАННЫЕ (ПОЛЕ) | ТИП | ПРАВИЛО ПРЕОБРАЗОВАНИЯ | ИТОГОВЫЕ ДАННЫЕ (ПОЛЕ) | ТИП |
| :--- | :--- | :--- | :--- | :--- |
| next_billing_date | дата-время | ЕСЛИ оплата успешна ТО прибавить `period_months` месяцев | next_billing_date | дата-время |
| status | перечисление | ЕСЛИ оплата НЕ успешна ТО сменить "active" на "paused" | status | перечисление |
| price | дробное | Записать сумму фактического списания в платеж | amount | дробное |
| системный ответ шлюза | строка | ЕСЛИ успех ТО "paid" ИНАЧЕ "failed" | payment_status | перечисление |
