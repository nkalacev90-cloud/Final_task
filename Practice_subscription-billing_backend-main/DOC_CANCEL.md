# Сценарий: Отмена подписки клиентом

## 1. АЛГОРИТМ РАБОТЫ

1. Получить `subscription_id` от клиентского приложения через PUT-запрос на эндпоинт `/subscriptions/{subscription_id}/cancel`.

2. Проверить существование подписки и права доступа:
   - ЕСЛИ подписка не найдена в БД ИЛИ её `user_id` не совпадает с ID текущего авторизованного пользователя, ТО вернуть ошибку с кодом 404 "Подписка не найдена".
   - ИНАЧЕ продолжить выполнение.

3. Проверить текущий статус подписки:
   - ЕСЛИ статус подписки = "active" (активна) ИЛИ "trial" (пробная), ТО изменить статус на "cancelled" (отменена).
   - ИНАЧЕ вернуть ошибку с кодом 400 "Подписка уже неактивна или отменена".

4. Сформировать итоговые данные подписки по правилам части «Преобразования».

5. Вернуть клиенту обновленный объект подписки и дату окончания доступа (`end_date`).

## 2. ТИПЫ ДАННЫХ

| № | НАЗВАНИЕ ПОЛЯ | ТИП |
| :--- | :--- | :--- |
| 01 | subscription_id | целое |
| 02 | user_id (из токена) | целое |
| 03 | status | перечисление (active, trial, cancelled, paused, expired) |
| 04 | end_date | дата-время |

## 3. ПРЕОБРАЗОВАНИЯ

| ИСХОДНЫЕ ДАННЫЕ (ПОЛЕ) | ТИП | ПРАВИЛО ПРЕОБРАЗОВАНИЯ | ИТОГОВЫЕ ДАННЫЕ (ПОЛЕ) | ТИП |
| :--- | :--- | :--- | :--- | :--- |
| status | перечисление | ЕСЛИ статус был "active" или "trial", ТО присвоить "cancelled" | status | перечисление |
| end_date | дата-время | Оставить без изменений (доступ сохраняется до конца оплаченного периода) | end_date | дата-время |
